Follow this exact sequence when `/subir-datos-databricks` is invoked. It is the standalone, single-source-of-truth command for mirroring local `data/` into the Unity Catalog Volume `workspace.default.chec-simulador` — useful on its own when only the raw/data files changed and neither the source packages, the `project_flow` notebooks, the tables/dashboard, nor the reports need to be touched. `/subir-a-databricks` reuses this exact command by cross-reference instead of duplicating this logic.

**Out of scope**: this command MUST NOT create or refresh any Delta table, MUST NOT run any job or notebook, MUST NOT touch the Lakeview dashboard, and MUST NOT upload anything under `notebooks/`, `src/`, or `reports/`. It only mirrors files into the Volume's `data/` folder.

## 0. Ask the user for the one required input

Ask, and wait for the answer:
1. The Databricks workspace URL (e.g. `https://dbc-xxxxxxxx-xxxx.cloud.databricks.com`).

Do not ask about profile or catalog up front — resolve those automatically below.

## 1. Resolve a CLI profile for that workspace (reuse, do not re-derive)

Follow `.claude/commands/deploy-databricks-dashboard.md` **section 1** ("Resolve a CLI profile for that workspace") verbatim, using the workspace URL from step 0. Carry forward the resolved `<profile>` for every command below.

- Matching profile found → use it.
- No matching profile → tell the user to run `databricks auth login --host <workspace-url>` via the `!` prefix (interactive OAuth, cannot be run for them), then re-resolve.

No warehouse resolution is needed — this command never runs SQL.

## 2. Upload the `data/` tree to the Volume

Mirror the full local `data/` directory to the Volume as-is — no format conversion, no Delta-table creation here (that stays the job of `notebooks/databricks/uiti_vano_tables.py`, reused unmodified by `/deploy-databricks-dashboard` and `/subir-a-databricks`):

```
databricks fs cp -r data dbfs:/Volumes/workspace/default/chec-simulador/data --overwrite -p <profile>
```

Exclude before uploading (do not copy these): `.DS_Store`, `.gitkeep`, `.openmeteo_cache.sqlite` (regenerable HTTP cache, not source data).

Document the hybrid nature of this upload to the user:

| File / pattern | Treatment |
|---|---|
| `Indicadores_vano_v3.csv` | Uploaded as-is; also the sole Delta-conversion candidate materialized elsewhere (by `uiti_vano_tables.py`, not by this command) into `indicadores_vano` / `circuit_clustering` / `circuit_geo` |
| `GEO/*` (all shapefile sidecars) | Uploaded as-is, binary, no conversion |
| `graphs/*.npy`, `graphs/*.json` | Uploaded as-is |
| `models/*.zip`, `models/manifest.sha256.json` | Uploaded as-is |
| `optuna/*.journal`, `optuna/*.pkl` | Uploaded as-is |
| `COSTOS ITEMS CONTRATOS.xlsx`, `Variables_seleccion.xlsx` | Uploaded as-is |

## 3. Upload `site/data/variables.json` as an extra data item

This is the one file this command uploads from outside `data/`, because `notebooks/project_flow/07_graph_preserved_connections_uiti_vano.ipynb` needs it as a required read-only input when it later runs in Databricks (see `/subir-notebooks-databricks`), and `site/` is otherwise never migrated (its HTML/PNG assets are only ever regenerated by a local run against the real repo paths — see the site-scope note in `/subir-a-databricks` and `/subir-notebooks-databricks`):

```
databricks fs cp site/data/variables.json dbfs:/Volumes/workspace/default/chec-simulador/data/variables.json --overwrite -p <profile>
```

This is a plain data-file upload, not a `site/` folder mirror — nothing else under `site/` is touched by this command, and this command never writes anything back into the Volume under a `site`-named path.

## 4. Report back

Tell the user, in their language:
- The profile and workspace this run used.
- Confirmation that `data/` (minus the excluded files) and `site/data/variables.json` were uploaded to `dbfs:/Volumes/workspace/default/chec-simulador/data`.
- That no tables, notebooks, or dashboard were touched by this run — if those are also needed, point to `/subir-notebooks-databricks`, `/subir-a-databricks`, or `/deploy-databricks-dashboard` as appropriate.
